## RESEARCH FINDINGS (from Anthropic docs)

**How prompt caching works:**
- Add `"cache_control": {"type": "ephemeral"}` to content blocks
- System checks if prefix (up to breakpoint) is already cached
- 5-minute TTL default (refreshed on use), 1-hour option at 2x cost

**Pricing (Opus 4.5):**
- Base input: $5/MTok
- Cache WRITE (5m): $6.25/MTok (1.25x)
- Cache READ: $0.50/MTok (0.1x) — **90% discount!**

**Cache hierarchy:** `tools` → `system` → `messages`

**Minimum cacheable tokens:**
- Opus 4.5: 4096 tokens
- Opus 4/4.1, Sonnet 4/4.5: 1024 tokens

**What invalidates cache:**
- ANY change to content before breakpoint
- Tool definition changes invalidate everything
- Image presence/absence changes
- Extended thinking parameter changes

**Key implementation details:**
- Up to 4 cache breakpoints allowed
- 20-block lookback window for automatic prefix checking
- Thinking blocks can't have cache_control directly but get cached alongside other content

## ANALYSIS IN PROGRESS

**Current cache implementation (anthropic_client.py):**

Three breakpoints set:
1. **Tools** (line 490): Last tool gets `cache_control: ephemeral`
2. **System message** (line 497): `_add_cache_control_to_system_message()` — wraps entire system as one cached block
3. **Final message** (line 534): `_add_cache_control_to_messages()` — marks last content block

**Potential issue identified:**
Current impl caches the ENTIRE system message as one block. But system message likely contains:
- Static portion (core memories, persona, instructions) — STABLE
- Dynamic portion (compaction summary, recent context) — CHANGES OFTEN

If mixed together, ANY change to dynamic portion invalidates cache for entire system message.

**Potential optimization:**
Split system message into multiple content blocks:
1. Static content → cache_control: ephemeral
2. Dynamic content → no cache_control (or separate breakpoint)

## ARCHITECTURE UNDERSTANDING (pieced together)

**Flow: How system message gets built**

1. `letta_agent.py` calls `_rebuild_memory_async()` (from base_agent.py)
2. `base_agent.py._rebuild_memory_async()` calls `agent_state.memory.compile()`
3. `schemas.memory.py.compile()` renders:
   - `<memory_blocks>` (persona, human, etc. — line-numbered for Anthropic)
   - Tool usage rules
   - `<directories>` (sources/files)
4. `PromptGenerator.get_system_message_from_compiled_memory()` combines:
   - Base system prompt (`agent_state.system`)
   - Compiled memory string
   - Metadata (timestamps, counts, archive tags)
5. Result becomes single system message → sent to Anthropic

**Current caching problem:**
ALL of this is wrapped as ONE content block with cache_control. So ANY change invalidates everything:
- Memory block edit? Cache miss.
- New message added? Cache miss (if summary changes).
- Timestamp change? Cache miss.

**Potential optimization strategy:**
Split system message into multiple content blocks:
```
[
  {"type": "text", "text": "<base_prompt>", "cache_control": {"type": "ephemeral"}},
  {"type": "text", "text": "<memory_blocks>", "cache_control": {"type": "ephemeral"}},
  {"type": "text", "text": "<dynamic_metadata>"}  // no cache_control
]
```

This would require:
1. Changes to `compile()` or new method that returns parts separately
2. Changes to `anthropic_client.py` to handle multi-part system messages
3. Changes to `base_agent.py` to assemble parts with proper cache markers

**Complexity assessment:** This is NOT a simple change. Multiple files, architectural understanding needed.

## WHAT COUNTS AS "STATIC" vs "DYNAMIC"

**STATIC (cache these):**
- Base system prompt (`agent_state.system`) — rarely changes
- Memory blocks (`<memory_blocks>`) — changes only on explicit edits
- Tool definitions — already cached separately

**DYNAMIC (don't cache, or separate breakpoint):**
- Timestamps (`in_context_memory_last_edit`)
- Message counts (`previous_message_count`)
- Archival memory size
- Archive tags list (changes as tags are added)
